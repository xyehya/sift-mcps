# AgentIR — Architecture

## Overview

AgentIR is a two-machine architecture: a SIFT Workstation VM running the forensics tooling and gateway, and a separate analyst machine running the AI agent (Hermes). The two communicate over HTTPS only. The SIFT VM is never reachable via SSH from the agent.

```
┌──────────────────────────────────────────────────────────────────────────┐
│ SIFT VM                                                                  │
│                                                                          │
│  sift-gateway  (Starlette ASGI, :4508 HTTPS)                            │
│  ┌────────────────────────────────────────────────────────────────────┐  │
│  │ Middleware Stack (every request)                                   │  │
│  │  AuthMiddleware → RateLimitMiddleware → SecureHeadersMiddleware    │  │
│  └────────────────────────────────────────────────────────────────────┘  │
│                                                                          │
│  ┌─────────────────────────┐   ┌──────────────────────────────────────┐  │
```

## Component Breakdown

### sift-gateway

The single HTTPS entry point for everything. Handles:

- **TLS termination** — self-signed CA generated by the installer; one keypair per installation
- **Bearer token authentication** — `AuthMiddleware` validates `agentir_gw_*` (examiner) and `agentir_svc_*` (service/agent) tokens; tokens expire
- **Rate limiting** — per-principal sliding window; examiner lockout after repeated failures
- **Secure HTTP headers** — HSTS, CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy
- **MCP aggregation** — proxies tool calls to stdio backends; builds a unified tool namespace
- **Evidence gate** — pre-routing chain-of-custody verification (see [Evidence Chain of Custody](evidence-chain-of-custody.md))
- **Response guard** — scans and redacts secrets/PII from tool output before it reaches the agent
- **Audit envelope** — writes a structured JSONL entry for every `/mcp` tool call regardless of outcome

### case-dashboard (Examiner Portal)

A Starlette sub-app mounted at `/portal/` providing:

- **Session management** — JWT cookies with sliding refresh, JTI revocation, login rate limiting
- **Must-reset guard** — blocks all write operations until the default password is changed
- **RBAC** — examiner vs agent roles; each endpoint declares minimum required role
- **Case creation** — creates the directory, `CASE.yaml`, and all canonical files; activates the gateway case
- **Evidence intake workflow** — scan, register, seal, ignore, retire with HMAC password confirmation
- **Findings/timeline review queue** — agent-proposed items enter here; examiner confirms or rejects
- **Chain-of-custody views** — ledger events, anchor status, immutable flag status, HMAC verify

### agentir-core

The shared library that owns all forensic logic:

- Evidence manifest and ledger operations (`evidence_chain.py`)
- PBKDF2 key derivation (`derive_auth_key`, `derive_ledger_key` — domain-separated)
- HMAC challenge-response (`approval_auth.py`)
- Case I/O helpers (`case_io.py`)
- Solana SPL Memo anchoring (`anchor_manifest`, `load_anchor_proof`)
- OS-level immutability (`_set_immutable`, `get_immutable_flag`)

No other package duplicates this logic. `sift-gateway`, `case-dashboard`, and `case-mcp` all import from `agentir-core`.

### FastMCP stdio backends

All eight backends run as stdio subprocesses. The gateway spawns them on startup and routes tool calls via stdin/stdout JSON. They inherit `AGENTIR_*` environment variables (including `AGENTIR_CASE_DIR`) from the gateway process.

| Backend | Primary capability |
|---------|-------------------|
| `forensic-mcp` | Forensic enrichment, hash lookups, pattern matching |
| `case-mcp` | Case ops: case_status, evidence_list, evidence_verify (read-only hint) |
| `sift-mcp` | Sandboxed command execution (`shell=False`, allow-list, output limits) |
| `report-mcp` | Signed case report generation; embeds evidence chain state |
| `opensearch-mcp` | Elasticsearch-compatible artifact indexing and search |
| `forensic-rag-mcp` | Semantic DFIR context via vector search |
| `windows-triage-mcp` | Known-good baseline lookups (SQLite, offline-capable) |
| `opencti-mcp` | CTI enrichment (optional, requires OpenCTI endpoint) |

## Request Lifecycle — Agent Tool Call

```
1. Hermes sends HTTP POST /mcp with Bearer agentir_svc_* token

2. AuthMiddleware
   └── Validates token signature, expiry, and role
   └── Extracts examiner identity, role, token_id (SHA-256 fingerprint)
   └── Rejects unknown or expired tokens with 401

3. RateLimitMiddleware
   └── Per-principal sliding window check
   └── 429 after threshold; examiner lockout state persisted to disk

4. MCP endpoint — _call_tool()
   └── Evidence gate (evidence_gate.check_evidence_gate)
       ├── ACTIVE case + SEALED manifest + clean diff → continue
       ├── UNSEALED + readOnlyHint=true → inject warning, continue
       ├── UNSEALED + analysis tool → block, return structured error
       └── Any violation (MODIFIED/MISSING/UNREGISTERED/LEDGER_ERROR) → block all
   └── Route to backend via stdio MCP proxy
   └── Response guard (response_guard.redact_tool_result)
       ├── Scan 25 patterns: API keys, tokens, SSH keys, PII, credentials
       ├── Redact critical+high inline → [REDACTED:pattern_name]
       └── Flag medium/low with _agentir_context.secret_warning
   └── Audit envelope (sift-gateway.jsonl)
       └── Writes: ts, examiner, role, token_id, source_ip, backend, tool,
                   status, elapsed_ms, evidence_gate_result, redacted_patterns

5. Structured response returned to Hermes
```

## Request Lifecycle — Examiner Evidence Seal

```
1. Examiner visits portal, opens evidence intake panel

2. Portal scans evidence/ against sealed manifest
   └── Reports: unregistered, missing, modified, ok counts

3. Examiner requests HMAC challenge
   └── POST /api/evidence/chain/challenge
   └── Server generates nonce, binds to examiner + IP + 5-minute TTL

4. Examiner provides password — browser computes HMAC(stored_hash, nonce)
   └── POST /api/evidence/chain/seal {examiner, challenge_id, response_hmac}
   └── Server re-derives HMAC from stored PBKDF2 hash, compares — constant-time
   └── Derives ledger signing key: derive_ledger_key(stored_hash_hex)

5. seal_manifest() runs
   └── For each file: clear +i flag, compute SHA-256, verify path in evidence/
   └── Write new manifest version N+1 (previous_manifest_hash chain link)
   └── Append MANIFEST_SEALED event to evidence-ledger.jsonl (HMAC-signed, fsync)
   └── Set +i flag on each file (requires CAP_LINUX_IMMUTABLE)
   └── Chmod ledger 0444

6. Auto-anchor (if AGENTIR_SOLANA_KEYPAIR set)
   └── anchor_manifest() → SPL Memo tx payload: AGENTIR|{manifest_hash[:16]}|{ledger_tip[:16]}
   └── POST to Solana JSON-RPC (devnet/mainnet)
   └── Write evidence-anchor-v{N}.json with tx signature + explorer URL

7. Response guard override invalidated (evidence chain mutated)
   └── Gateway evidence gate cache invalidated immediately via on_chain_mutation callback
   └── inotify watcher also triggers cache invalidation on any evidence/ filesystem event
```

## Data at Rest

| File | Location | Protection |
|------|----------|------------|
| Evidence files | `/cases/{id}/evidence/` | chattr +i, AppArmor DENY write |
| Evidence manifest | `/cases/{id}/evidence-manifest.json` | chmod 0444 after write |
| Evidence ledger | `/cases/{id}/evidence-ledger.jsonl` | HMAC-signed events, chmod 0444 |
| Anchor proof | `/cases/{id}/evidence-anchor-v{N}.json` | Solana tx signature |
| Approvals | `/cases/{id}/approvals.jsonl` | Append-only, HMAC-signed |
| Examiner passwords | `/var/lib/agentir/passwords/` | PBKDF2-HMAC-SHA256, 600k iterations |
| Service tokens | `/var/lib/agentir/tokens/` | chmod 600, owner-readable only |
| TLS private key | `~/.agentir/` | chmod 600 |
| Audit logs | `/cases/{id}/audit/*.jsonl` | Written by gateway at tool-call time |
| Kernel audit | `/var/log/audit/audit.log` | auditd daemon, root-only write |

## Deployment

The entire SIFT VM side is deployed by a single `install.sh` script that:

1. Validates OS and Python runtime prerequisites
2. Installs uv and syncs all Python packages
3. Creates state directories (`/var/lib/agentir/`, `/cases/`)
4. Downloads forensic triage databases (optional)
5. Generates TLS certificate authority and keypair
6. Writes `~/.agentir/gateway.yaml` from template
7. Creates the default examiner account with PBKDF2-hashed temporary password
8. Configures OpenSearch Docker (optional)
9. Installs and enables the systemd user service
10. Sets `cap_linux_immutable+ep` on the uv-managed Python binary
11. Installs auditd rules for evidence write/attribute events
12. Loads AppArmor profile in complain mode
13. Polls the gateway health endpoint and prints operator handoff material
