System in one paragraph

The Gateway is the single policy boundary: every REST call, every MCP tool call, every privileged action passes through it. Postgres is the authoritative control plane (RLS-forced); OpenSearch is derived and never authoritative. AI agents reach tools only through the aggregate /mcp; humans use the React portal at /portal. Heavy work runs as durable Postgres jobs claimed by least-privilege workers, and deep execution is confined to an OS-sandboxed run_command plane.

VP-1 C4 Model · Level 1 — System Context

Who and what the gateway serves

Addresses the concern of scope & external interfaces: the system is a single trust-bearing unit with exactly two classes of consumer and one external dependency. There are no other ingress paths.

Reading it: only two consumer classes exist — a human operator over HTTPS to /portal, and an AI agent over MCP to /mcp. Both terminate at one system. The lone external dependency, OpenCTI, is reached through the gateway under a query-only contract — the agent never talks to it directly. This is the whole point of the design: there is no second door.

VP-2 C4 Model · Level 2 — Containers / 8 Planes

The eight planes and the one gate between them

Addresses decomposition & authority: how runnable units split into planes, which plane is the source of truth, and the fact that every path from a client to anything privileged is funneled through a single gold band. Color encodes plane authority (see key).

Reading it: the gold band is the only thing both clients can reach, and everything below it is reachable only from inside that band. Authority flows one way: Postgres (blue) is the source of truth; OpenSearch (teal) is a derived projection that is rebuilt from artifacts and is never trusted as the system of record. Heavy work doesn't block the agent — the gateway enqueues a durable job, least-privilege workers claim it under a lease, and confined execution writes results back up to Postgres and out to the derived index. Evidence (green) is operator-mounted and immutable; reports only ever contain approved material.

VP-3 UML 2.5 · Sequence — Tool-Call Policy Chain

One ordered path, nine fail-closed gates

Addresses runtime security behavior: the exact ordered chain every agent tool call traverses (verified in mcp_server.py + policy_middleware.py). A deny at any stage short-circuits to an audited MCP error — the tool body never executes. Projected as a gated pipeline for legibility; message order is preserved.

Reading it: identity is resolved before the chain even begins, and stages 1–9 run in fixed order. Five of them can reject outright (dashed red), and stage 7 — the EvidenceGate — is the hard interlock: no tool touches a case until its evidence is registered, sealed, and the custody chain verifies. Anything that escapes a gate lands on the single audited deny rail; only a call that clears all nine reaches the tool body, and even its output is redacted on the way back out.

Reading it: the two boundaries a forensic system lives or dies on are #2 (evidence can never be silently mutated — physical immutability plus a verifying chain) and #6 (tool output is treated as hostile and scrubbed before it re-enters the agent's context — the prompt-injection-from-evidence defense). Boundary #4 encodes the authority rule from VP-2: even a fully-authorized write lands in Postgres under RLS, never in the derived index.

VP-5 UML Deployment + Defense-in-Depth — run_command

The jail every forensic command falls through

Addresses execution containment & deployment: where confined work runs (the SIFT VM node, as a dedicated runtime user) and the two stacked layers — a policy ceiling and a kernel floor — that a command must clear before a forensic binary ever executes. Both default to deny.

Reading it: the ceiling reasons about intent — is this program allowed, does its text smuggle a shell, does the environment leak an interpreter — and the floor enforces capability at the kernel: even a command that talked its way past policy can't read outside the case paths (Landlock), can't make a disallowed syscall (seccomp kills it), can't reach the network or exceed its memory (cgroup), and can't regain privilege (no-new-privs). The defaults are deny on both layers, which is why the red-team rows fail without ever asking for approval.
