## 1. Executive Summary

PR01 should implement the first feature-bearing migration slice: Phase ID-1 from
`09_identity_auth_cutover.md`, the control-plane identity schema foundation.

This PR should add the initial Supabase/Postgres schema objects needed for the
cases/tokens/identity cutover order:

- `app.operator_profiles`
- `app.cases`
- `app.case_members`
- `app.active_case_state`
- `app.agents`
- `app.service_identities`
- `app.audit_events`
- `app.mcp_tokens`
- `app.mcp_token_scopes`

The PR is schema-first and intentionally does **not** connect the Gateway,
portal, MCP tools, workers, or OpenSearch runtime paths to the new tables. The
goal is to create the stable DB surface that later PRs can wire into without
mixing schema design with behavioral auth changes.

This PR must be:

- Additive only.
- Schema and schema tests only.
- Compatible with Supabase Local/Postgres.
- Aligned with charter decisions D4, D8, D10, D11, D12, and D17.
- No runtime behavior changes.
- No token validation changes.
- No portal auth changes.
- No Gateway auth or case-context propagation changes.
- No job tables, workers, OpenSearch refactors, evidence migrations, or frontend
  changes.

## 2. Why This PR Comes First

The charter locks migration cutover order as cases/tokens/identity first. JOB-0
created a protective baseline without feature-bearing changes; PR01 is the first
actual cutover slice.

Durable jobs, evidence metadata, OpenSearch index registration, findings,
reports, RAG, and skills all require case and identity primitives. Adding those
systems before `cases`, `operator_profiles`, `case_members`, MCP/service token
tables, and active-case state would force them to depend on today’s file/env
authority, which the migration explicitly retires.

PR01 should happen before token runtime changes because the Gateway cannot
dual-validate DB-backed tokens until the hash-only registry tables and scope
tables exist.

PR01 should happen before Supabase human-auth wiring because session validation
needs `operator_profiles` and `case_members` as the local authorization model.

PR01 should happen before control-plane active-case propagation because the
authoritative active case row and case table need to exist before the Gateway
can read and propagate them.

PR01 should happen before jobs because every job row later needs immutable
`case_id`, `requested_by_user_id`, and/or `requested_by_token_id` context.

## 3. Exact PR Scope

The PR should include only:

1. Identify the existing database/migration layout and schema test pattern, or
   create the smallest conventional Supabase migration/test layout if none
   exists.
2. Add SQL migrations for the Phase ID-1 identity foundation tables.
3. Add deterministic schema tests that apply the migration to an isolated local
   test database or validate the SQL structure with the repository’s existing
   migration test mechanism.
4. Add a short documentation note explaining how to run the PR01 schema checks.
5. Avoid any production runtime behavior change.

The PR may include:

- `app` schema creation and required extensions such as UUID generation if the
  existing migration layout does not already provide them.
- Table DDL, primary keys, foreign keys, indexes, check constraints, comments,
  and RLS enablement.
- Minimal seed-safe enum/check values for statuses and roles.
- Service-safe helper comments or views only if already established by local
  schema patterns.
- Tests that assert table existence, key constraints, no plaintext-token
  columns, RLS enabled where applicable, and role/status constraints.

The PR must not include:

- Gateway token validation or `verify_api_key()` changes.
- Portal login/session/Supabase Auth integration.
- Token creation, rotation, revocation, or UI changes.
- Active-case runtime propagation or compatibility exports.
- Evidence gate changes.
- REST job APIs, durable jobs, workers, job tables, job tools, or job status UI.
- OpenSearch runtime changes, parser changes, or index-registration runtime.
- Evidence/audit data migration from JSONL/manifest files.
- Frontend changes.
- Docker/OpenSearch deployment changes.
- Cleanup of unrelated `.DS_Store`, config, generated, or local-state changes.

## 10. Rollback Strategy

Because PR01 is schema-only and additive, rollback should be straightforward:

- Revert the migration/test/docs commit if it has not been applied to shared
  environments.
- In local development databases, drop the added migration or reset the local
  Supabase/Postgres database.
- If the migration was applied to a shared dev database, add a follow-up
  rollback migration that drops the new PR01 objects in dependency order.

Rollback must not touch existing file-backed evidence, audit JSONL, Gateway
config, active-case pointer files, OpenSearch indices, or case directories.

## 13. Decisions And Open Questions

Locked decisions for PR01:

- Use `app` schema for authority tables, with service-only helper schema only if
  the implementation needs it.
- Use UUID primary keys plus explicit legacy text keys.
- Store MCP/service tokens hash-only; raw tokens are never stored.
- Initial roles are `readonly`, `operator`, `lead`, `owner`, and `admin`.
- Active case is one `deployment`-scope row in v1.
- Runtime dual-validation and Supabase Auth wiring are later PRs, not PR01.

Open questions for the coding run:

- Does the repository already have a Supabase migration directory? If not,
  choose the smallest conventional layout and document it.
- Is there an existing local Postgres/Supabase test harness? If not, use SQL
  structure tests for PR01 and record the DB-harness gap.
- Should SQL enum types or check constraints represent statuses/roles in the
  first migration? Prefer check constraints unless the existing schema pattern
  clearly favors enums.
- Which exact RLS policies should be included in PR01 versus deferred to the
  runtime auth PR? At minimum, RLS should be enabled and unsafe direct access
  avoided; detailed policies can be narrow if runtime auth context is not wired
  yet.
