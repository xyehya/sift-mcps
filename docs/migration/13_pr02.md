# PR02 Implementation Candidate

Last updated: 2026-06-07.

Scope: planning only. This document defines the next implementation PR
candidate after PR01 identity foundation schema. It does not implement code,
create runtime token validation, refactor portal auth, or start the PR.

Candidate PR title:

> Add hash-only MCP token registry validation with legacy gateway.yaml fallback.

## 1. Executive Summary

PR02 should implement Phase ID-2 from `09_identity_auth_cutover.md`: the
hash-only MCP/service token registry runtime path.

PR01 created the foundational tables:

- `app.mcp_tokens`
- `app.mcp_token_scopes`
- `app.agents`
- `app.service_identities`
- `app.audit_events`
- supporting case and operator identity tables

PR02 should make the Gateway validate MCP/service tokens against
`app.mcp_tokens.token_hash` first, using the locked v1 token policy:

- `token_hash = sha256(server_pepper || token)`
- `token_fingerprint = first 16 hex chars of sha256(token)`
- raw token material is never stored
- active, unexpired, unrevoked token rows are required
- tool/action scopes come from `app.mcp_token_scopes`
- legacy `gateway.yaml api_keys` validation remains as a read-only fallback
  during cutover

PR02 may also move token lifecycle writes for create/rotate/revoke/reactivate
from `gateway.yaml` mutation to Gateway/portal service paths that write the DB
registry and display the raw token exactly once. Keep that work narrowly scoped
to backend/service behavior and tests; do not redesign the frontend.

This PR must be:

- Additive and reversible.
- Gateway token-validation focused.
- Hash-only for new DB-backed tokens.
- Compatible with legacy raw tokens during the cutover window.
- Covered by deterministic tests that do not require OpenSearch, forensic
  samples, workers, job tables, active-case propagation, or parser execution.

This PR must not include:

- Supabase human-auth/session integration.
- Case-membership authorization for humans beyond what token lookup needs.
- Active-case runtime propagation or compatibility exports.
- Evidence-gate case migration.
- Job tables, workers, REST job APIs, MCP job tools, or job status UI.
- OpenSearch runtime changes.
- Parser changes.
- Evidence behavior changes.
- Audit JSONL/control-plane data migration.
- Frontend redesigns or unrelated portal view changes.
- Removal of legacy `gateway.yaml api_keys` validation.

## 2. Why This PR Follows PR01

The charter locks the cutover order as cases/tokens/identity first. PR01 added
the database surface required for tokens; PR02 is the first runtime use of that
surface.

PR02 should happen before Supabase human auth because MCP/service clients are
not Supabase users and need the DB-backed registry path independently.

PR02 should happen before active-case propagation because the Gateway must know
which token is valid, unexpired, unrevoked, and scoped before it can safely
combine token scope with active-case context.

PR02 should happen before jobs because future job rows need immutable
`requested_by_token_id` and token/tool scope context captured at enqueue time.

PR02 should keep the legacy fallback so operators and agents are not locked out
while existing `gateway.yaml` tokens are re-issued as DB-backed tokens.

## 3. Exact PR Scope

The PR should include only:

1. Identify the existing Gateway token validation, identity resolution, token
   generation, config, and portal token lifecycle paths.
2. Add a small DB token-registry access layer for Supabase/Postgres.
3. Add token hashing and fingerprint helpers that implement the locked policy.
4. Update Gateway MCP/service token validation to check DB-backed tokens first
   and legacy `gateway.yaml api_keys` second.
5. Ensure DB-backed validation enforces status, expiry, revocation, and at least
   one matching scope before authorizing privileged tool use.
6. Add deterministic tests for hash-only behavior, legacy fallback, expiry,
   revocation, scope loading, and raw-token non-persistence.
7. If included, update portal token lifecycle service paths so new token writes
   go to the DB registry and show raw token material exactly once.
8. Add a short PR02 runbook for VM/Supabase test commands.

The PR may include:

- A small Gateway config addition for the token pepper source, failing closed
  for DB token validation if it is missing.
- A minimal Postgres client abstraction or repository helper, following local
  patterns.
- Targeted audit event writes for token validation success/denial if they can be
  implemented without migrating existing audit JSONL behavior.
- A migration only if a small additive constraint/index is discovered to be
  required by the validation path. Do not add new job/evidence/OpenSearch
  tables.
- A test fixture that applies the PR01 schema to the VM Supabase/Postgres stack
  inside a rollback transaction.

The PR must not include:

- `operator_profiles` or `case_members` Supabase human-auth runtime wiring.
- Portal login/session replacement.
- Active-case DB reads, writes, propagation, or generated pointer exports.
- Evidence gate changes.
- Durable job implementation.
- Worker runtime.
- OpenSearch, parser, or evidence pipeline behavior.
- Token registry cleanup that removes the legacy fallback.

## 8. Manual Validation Steps

After automated tests pass:

1. Confirm new DB token rows contain `token_hash` and `token_fingerprint`, never
   raw token material.
2. Confirm the pepper is sourced from VM/runtime secret config and is not written
   to repository files.
3. Confirm legacy `gateway.yaml api_keys` validation remains read-only fallback.
4. Confirm a token with no scopes can do nothing.
5. Confirm revoked, expired, disabled, and missing-token paths fail closed.
6. Confirm portal human auth/session behavior is unchanged.
7. Confirm active case still comes from the legacy path until ID-4.
8. Confirm no job/evidence/OpenSearch/frontend changes are included.

## 9. Acceptance Criteria

The PR is acceptable when:

- DB-backed token validation checks `app.mcp_tokens.token_hash` before legacy
  config tokens.
- New DB-backed tokens are hash-only and peppered.
- Fingerprints are non-secret 16-hex correlation values.
- Expiry, revocation, disabled status, and missing scopes fail closed.
- Legacy `gateway.yaml api_keys` fallback remains available and covered by tests.
- Portal token lifecycle, if included, writes DB records and returns raw tokens
  exactly once.
- Targeted Gateway/portal token tests pass on the SIFT VM using Python 3.12.
- Optional live Supabase checks are rollback-safe.
- `git diff --check` passes.
- No Supabase human auth, active-case propagation, evidence gate, jobs, workers,
  OpenSearch, parser, evidence, or frontend changes are included.

## 10. Rollback Strategy

Because PR02 changes token runtime behavior, rollback should be explicit:

- Revert the PR02 runtime commit to restore legacy-only `gateway.yaml` token
  validation.
- Keep the PR01 schema in place unless the environment must be reset; the schema
  is additive and harmless without runtime use.
- If DB-backed token rows were created during testing, disable/revoke them rather
  than deleting audit-relevant records.
- Do not delete legacy `gateway.yaml api_keys` during PR02, so rollback does not
  require credential reconstruction.

Rollback must not touch active-case pointer files, evidence manifests, audit
JSONL files, OpenSearch indices, parser outputs, or case directories.

## 13. Decisions And Open Questions

Locked decisions for PR02:

- Store MCP/service tokens only as hashes.
- Use SHA-256 with a Gateway server pepper for `token_hash`.
- Use first 16 hex chars of SHA-256 over raw token as `token_fingerprint`.
- Display raw token material exactly once on creation/rotation.
- Validate DB registry first, then legacy `gateway.yaml api_keys` fallback.
- Legacy fallback is read-only and temporary.
- Revoked, expired, disabled, missing, and unscoped DB tokens fail closed.

Open questions for the coding run:

- Exact Postgres client/dependency pattern for Gateway service-role access.
- Exact token pepper source and config name in the VM/systemd environment.
- Whether portal token lifecycle should be included in the same PR or split
  into PR02b after Gateway validation is proven.
- Whether token validation audit events should be written in PR02 or limited to
  tests/stubs until the broader audit DB write policy is implemented.
- How much of `Identity` should be extended now versus deferred until active
  case and Supabase human auth phases.
