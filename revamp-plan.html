SIFT Protocol Gateway — Implementation Spec

This is the build spec for consolidating today's eight-subprocess gateway into a self-contained SIFT Protocol Gateway: a core (backed by sift-core, renamed from agentir-core) that owns execution, case management, evidence & chain-of-custody, audit, findings, reporting, the portal, and a /skills distribution endpoint that host skills required for an autonomous DFIR Agent running outside of SIFT VM to operatre — surrounded by optional add-on MCP backends governed by a strict, versioned connection contract.

Docs-only artifact; no code changed.

§Executive Summary

The current platform already contains every capability it needs and is well-tested. The problem is topology, not features: the capabilities we consider "core" (execute, case, evidence/chain-of-custody, audit, findings, reporting) run as independent stdio subprocesses that the gateway merely aggregates. The gateway is a router, not a core — if a core subprocess fails to boot, a core capability silently disappears. Case/findings logic is implemented across forensic-mcp, case-mcp and report-mcp in parallel with agentir-core; that duplication is the interoperability hazard we set out to eliminate.

The backend-connection mechanism already exists (sift_gateway/backends: an MCPBackend base, stdio + HTTP transports, a tool-map with reactive collision-prefixing) — but it is undocumented and unenforced. There is no manifest, no capability declaration, no health/identity/evidence-gate contract a third party could implement against, and no notion of core vs optional in config. We also observed live tool drift: the running gateway advertises ~62 tools while the source codes more than that (forensic-mcp alone codes 20 tools + 14 resources, most not currently surfaced) — concrete proof that "what is exposed" is not pinned to "what is declared."

The revamp: (1) promote the renamed sift-core to the single source of truth and register core tools in-process; (2) keep run_command behind a hardened isolated executor with a denylist-default policy and a capabilities-first / sudoers-allowlist privileged path; (3) publish a versioned SIFT MCP Backend Contract (spec + JSON Schema + conformance) that every add-on satisfies; (4) retire the forensic methodology knowledge tools into a /skills corpus (post-MVP). The result is a standardized baseline the community can extend safely. 

The main pitch for the Hackathon is building the SFIT Protocol Gateway that sets the solid secure extensible stone for the critical DFIR requirements centrally so that governance, oversight, security, audit is defined once and can integrate with any future mcp backend through a spec which provides plug-and-play tools and extensions that smoothly integrate with the SIFT Protocal Gateway and sift-core.

✓Locked Decisions

Architecture (from v1 clarification)

CORE BOUNDARY

Consolidate in-process

sift-core becomes the single source of truth; the core backends collapse into the gateway as in-process tools. Add-ons stay external.

EXECUTE

Keep sandboxed

Core owns and provides the execute (run_command or execute tool) contract, policy, and audit; process spawning is delegated to a hardened isolated sandbox (separate process · cgroup · AppArmor · shell=False).

CONTRACT

Spec + schema + conformance

A versioned protocol spec, a machine-checkable manifest schema, and a conformance checklist an external team can implement against. This should be uncomplicated and clear; a simple pathway for developers to write and connect their mcp tools to extend current functionalities (e.g opensearch)

CONTEXT INJECTION

Split, don't centralize

Static agent guidance → manifest instructions, gateway-composed. Runtime response enrichment stays per-backend (declared capability). New backends must be able to enrich their own responses.

New decisions (2026-05-31)

D1 · RENAME

agentir → sift

Rename the Python package agentir_core → sift_core and the runtime env/path surface (AGENTIR_* → SIFT_*, /var/lib/agentir → /var/lib/sift), plus the agentir_svc_*→sift_svc_* token prefix. No back-compat — fresh VM + case, clean cutover (no shim, no symlink). See §6.

D2 · EXECUTE POLICY

Denylist-default

run_command stays denylist-based (its current real behaviour). The denylist is already externalized to security.yaml — the work is not "extract from hardcoded" but: (a) relocate it from the package's data/catalog/ to operator-editable gateway.yaml, (b) add a non-weakenable deny floor, (c) add allowlist mode. See §5.

D3 · PRIVILEGE

Caps-first, sudoers fallback

Executor runs unprivileged. Root-needing tools (vol, dd, mount) escalate via Linux capabilities first, falling back to a narrow sudoers allowlist (hashed paths, NOPASSWD, never a shell/wildcard). Every escalation audited. See §5.

D4 · METHODOLOGY

Remove tools → /skills

The 14 forensic methodology get_* tools + 14 resources are removed as MCP tools and served through /skills packs. Post-MVP — implemented last. See §7.

2.1 — Three tiers, not two

CORE — always present

Gateway-owned, backed by sift-core. Zero add-on dependency. Cannot be disabled.

  • Execute (sandboxed)
  • Case mgmt + metadata
  • Evidence & chain-of-custody
  • Findings · timeline · TODO
  • Reporting (signed)
  • Audit / reasoning / external-action
  • Surfaces: /mcp, /portal, /skills, health
  • Trust layer (auth, gate, guard)
  • All portal activities 
SKILLS — methodology corpus

Static, versioned, signed markdown packs served from /skills. Replaces the forensic-mcp methodology tools (D4). Post-MVP.

  • General DFIR pack
  • Per-discipline packs (windows, memory, network…)
  • Sourced from forensic-knowledge + ex-forensic-mcp resources + your hardened skills
  • Loaded once by the operator, not called per-tool
ADD-ON — optional

Installed per situation / hardware. Connects via the contract. Absence degrades gracefully.

  • opensearch-mcp (Docker + RAM)
  • opencti-mcp (OpenCTI + ≥14 GB)
  • forensic-rag-mcp (vector index)
  • windows-triage-mcp (SQLite, offline)
  • community backends
Graceful degradation: the gateway advertises an add-on's tools only when the backend is present and its declared requires[] are satisfied.

4SIFT MCP Backend Contract v1

The standard others build on. A backend satisfying this contract can be dropped on a SIFT VM and aggregated with no compatibility negotiation.

4.1 — Backend manifest (sift-backend.json)

Every backend ships a manifest at a well-known path. The gateway loads it at registration, validates against the schema, runs the conformance probe (reusing the /mcp/{name} endpoint, P8), and only then merges the backend's tools into /mcp. This replaces today's reactive discovery and collision-prefixing.

4.2 — Evidence-Gate spec (current → target · decision F-A)

What it is (the forensic-soundness interlock)

The gate runs on every /mcp call before the tool executes. It asks one question about the case's evidence chain — "is the chain in a trustworthy state?" — by calling chain_status(case_dir) from sift-core, cached 30 s (invalidated on manifest mtime change or explicit portal seal via invalidate_evidence_cache). Its purpose: make "you cannot analyze evidence that has not been cryptographically sealed and verified" a structural guarantee, so downstream findings are court-defensible.

Current behaviour (verified)

ChainStatusMeaningEffect today
OKEvidence registered, hashes verify, ledger intact, sealedEverything proceeds
UNSEALEDNo evidence registered yet — normal at case start, not a violationRead-only tools allowed (carve-out applied in endpoint via readOnlyHint); others blocked
MODIFIED / MISSING / UNREGISTERED / LEDGER_ERRORChain integrity violationEverything blocked, incl. read-only

So current model = case integrity status × tool readOnlyHint boolean. The boolean is too coarse and the endpoint must know tool names → community tools aren't gated automatically.

Target behaviour (decision F-A — strict binary, block everything)

Drop the readOnlyHint carve-out and the former 3-class model. The gate becomes one rule on the agent /mcp surface: if chain_status == OK (evidence registered and sealed) allow the tool; otherwise block every agent tool — including read-only — and return a portal-remediation response. Any non-OK status blocks until the operator fixes it in the portal and re-seals.

chain_statusEvery agent /mcp tool
OK (sealed)allow
UNSEALEDblock — nothing runs before the operator seals
MODIFIED / MISSING / UNREGISTERED / LEDGER_ERRORblock until fixed + re-sealed in the portal
Why simpler beats the 3-class target: "nothing runs against unsealed or compromised evidence" is the cleanest court-defensible custody invariant and the easiest to specify — a backend still declares readOnlyHint/mutating for audit/approval, but the gate itself needs zero per-tool classes and zero hardcoded tool names. Scope: the gate sits on the agent /mcp surface only; health/lifecycle/portal endpoints are not agent tools and stay ungated by design. This supersedes the former opt O2 3-class model and the readOnlyHint carve-out — the old "UNSEALED ≠ blocked for read-only" invariant no longer applies.

5Execute & Sandbox Spec (run_command)

The one tool that must never be a community-pluggable add-on — it is the blast-radius primitive. Core owns the contract, policy and audit; spawning is isolated; privilege is sanctioned and audited.

5.1 — Policy: denylist-default (D2)

  • Today (verified): shell=False + path-jail (resolve_case_path) + a denylist already loaded from sift-mcp/data/catalog/security.yaml via load_security_policy() (not hardcoded; loader refuses an empty policy). The list is categorized and broad: mkfs.*, shutdown/reboot/poweroff/halt/init, kill/killall/pkill, env/printenv (token-leak), raw sockets. The server.py docstring is stale and undercounts it. security.yaml also carries dangerous_flags, tool_allowed_flags, tool_blocked_flags, output_flags. Default cwd = AGENTIR_CASE_DIR.
  • Target: relocate the policy from the package's data/catalog/security.yaml to operator-editable config: execute.deny[] / execute.allow[] in gateway.yaml (keep the same schema: denied_binaries, dangerous_flags, per-tool allowed/blocked flags).
  • Deny floor: a hardcoded minimum denylist operators may extend but never weaken (you can't un-block mkfs/shutdown) — this is the one part that should NOT live only in YAML.
  • Optional allowlist mode: execute.mode: allowlist for high-assurance installs; default is denylist (DFIR needs a long tail of tools).
  • Keep: path-jail, shell=False, FK enrichment + skip_enrichment, save_output + SHA-256, input-file detection (llm → catalog → parsed fallback).

5.2 — Isolation (P6)

Extract spawning into a separate executor process the core calls over a narrow IPC. Boundaries: dedicated cgroup (CPU/mem/pids caps so an OOM/fork-bomb can't touch the gateway), dedicated AppArmor profile (see configs/apparmor/), shell=False, no network unless the command needs it. A crash/abuse in the executor must leave auth / evidence-gate / case state healthy. Audit still records every exec (the audit envelope stays in core).

5.3 — Privilege: capabilities-first, sudoers-allowlist fallback (D3)

vol / dd on a device / mount helpers need privilege, or artifacts can't be extracted. The executor runs unprivileged by default. Tools tagged requires_privilege: true (manifest, §4.1) escalate by this order:

  1. Linux capabilities (preferred): grant the executor the minimum caps for the read case — e.g. cap_dac_read_search (read any file for imaging), cap_sys_admin only where unavoidable. Avoids the sudo surface entirely for raw-read workloads.
  2. Sudoers allowlist (fallback): for genuine root-only tools, install.sh writes /etc/sudoers.d/sift-exec granting the executor user NOPASSWD on an explicit, full-path allowlist only (e.g. /usr/bin/vol, /usr/bin/dd, mount helpers). Never a shell, never a wildcard, never ALL.
Hard rules: the gateway never runs as root. No blanket passwordless sudo. The privileged path is reachable only for allowlisted, full-path binaries flagged requires_privilege, and every escalation is written to the audit envelope (which binary, why, by whom). The decision to escalate stays in core — backends cannot shell out to sudo themselves (conformance forbids direct shell).

Touches: configs/apparmor/* · configs/systemd/* · install.sh (sudoers.d + setcap) · sift-mcp/tools/generic.py · new executor service in sift-gateway or sift-core

10Invariants to Preserve & Risks

Non-negotiable invariants (every phase)

  • R1 — no direct shell; execution only via the sandboxed Execute service
  • R2 — atomic writes, SHA-256, append-only approvals, HMAC ledger, chattr +i
  • R3 — findings DRAFT until examiner HMAC approval
  • R4 — portal-created cases; case-dir env propagation (one source)
  • R5/R6 — TLS everywhere; timing-safe Bearer auth, two roles
  • R7 — aggregate /mcp is the only agent endpoint
  • R8 — gateway never root; privilege only via caps/sudoers-allowlist, audited
  • R9 — (F-A) strict binary evidence gate: no agent /mcp tool runs unless chain_status == OK; UNSEALED blocks everything until the operator seals (supersedes the old "UNSEALED ≠ violation" carve-out)
  • R10 — finding-time logic stays core, not a tool: validation (enforced in record_finding), Grounding Score, FK enrichment. None may be deleted when the 14 methodology tools become /skills.
  • R11 — zero hardcoded backend names in core logic: the Grounding Score must resolve reference backends from manifest provides:["reference"], mirroring the evidence gate's zero-tool-name rule.
  • R12 — operator drives the portal only: agent tokens are portal-minted (audit identity = agent_id, authorizer = created_by); the portal rejects agent tokens and /mcp is agent-only; human accountability attaches at DRAFT→approval.
  • R13 — (F-B) the agent (run_command jail = case dir) can never read or write the integrity records; they live at /var/lib/sift/<case_id>/.

Migration risks & mitigations

  • Rename breakage (D1) → central env helper; fresh VM means a clean cutover (no shim/symlink); full suite per phase; re-run discovery grep until grep -ri agentir packages/ is clean.
  • Behaviour drift in P1 → parity tests on case I/O before cut-over.
  • In-process core widens blast radius (P2) → only pure-library logic moves in; execution stays sandboxed.
  • Host vs SIFT VM (two OSes) → validate every phase on the VM, not just locally; caps/sudoers/apparmor are VM-specific.
  • Deleting methodology tools too early (D4) → sequenced last; audit runtime deps first.
  • Contract churn → freeze spec_version 1.0 before publishing the SDK.
